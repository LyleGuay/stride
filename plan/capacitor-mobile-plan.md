# Capacitor Mobile Setup

## Goal

Add Capacitor to `web-client/` so the existing React app can be built and installed as a native Android APK on a physical Pixel device. The core work is: wiring the API URL for native builds, preventing the PWA service worker from conflicting with the Capacitor WebView, creating a thin platform abstraction layer for native feature gating, fixing the calorie log's mobile layout (hiding desktop-only interactions), and adding a Playwright mobile-viewport project so responsive behavior is covered by the existing E2E suite.

This plan deliberately excludes native plugins (haptics, camera, notifications) — those are additive and can be tackled per-feature. The goal here is a solid, testable foundation.

---

## Phases

### Phase A: Capacitor Foundation

- [ ] **A.1 — Install Capacitor and initialize the Android platform**

  In `web-client/`:

  ```bash
  npm install @capacitor/core @capacitor/android
  npm install -D @capacitor/cli
  npx cap add android
  ```

  Create `web-client/capacitor.config.ts`:
  ```ts
  import { CapacitorConfig } from '@capacitor/cli'
  const config: CapacitorConfig = {
    appId: 'com.stride.app',
    appName: 'Stride',
    webDir: 'dist',
    server: {
      // Use https scheme so cookies and CSP behave consistently with the web app.
      androidScheme: 'https',
    },
  }
  export default config
  ```

  Update `CLAUDE.md` commands section with the Capacitor build workflow (added in A.3).

  - **Manual tests:** Open the generated `web-client/android/` folder in Android Studio on Windows (via `\\wsl$\...` path). Verify the project loads and Gradle syncs without errors. No app features need to work yet — just confirm the project is valid.

- [ ] **A.2 — Fix API URL so native WebView calls reach the Go API**

  The current `request()` in `web-client/src/api.ts` uses bare paths (`/api/...`). In the Capacitor WebView, those resolve to `https://localhost/api/...` — a dead end. The app needs to hit the real deployed API.

  In `web-client/src/api.ts`, read an env var as the base:
  ```ts
  // Empty string for web (relative paths still work via Vite proxy / Go static serving).
  // Set to the deployed API origin for Capacitor builds.
  const API_BASE = import.meta.env.VITE_API_BASE_URL ?? ''
  ```
  Prefix it in `request()`:
  ```ts
  const res = await fetch(API_BASE + path, { ...options, headers })
  ```

  Create `web-client/.env.capacitor.example` (committed):
  ```
  # Copy to .env.capacitor and fill in for Capacitor builds.
  # This is the deployed Go API origin — no trailing slash.
  VITE_API_BASE_URL=https://your-api.railway.app
  ```

  Add `web-client/.env.capacitor` to `web-client/.gitignore` (the actual file with secrets).

  - **Manual tests:** After A.3 (Capacitor build script), install on Pixel and verify login works, calorie log loads data from the real API.

- [ ] **A.3 — Separate PWA and Capacitor build modes**

  `vite-plugin-pwa` registers a service worker. Capacitor's WebView doesn't support service workers and the autoUpdate logic can interfere. Gate it:

  In `web-client/vite.config.ts`, wrap the `VitePWA(...)` plugin:
  ```ts
  // Only register the PWA service worker for web builds — Capacitor WebView
  // doesn't support service workers and the autoUpdate behaviour causes issues.
  ...(process.env.CAPACITOR_BUILD ? [] : [VitePWA({ ... })])
  ```

  Add scripts to `web-client/package.json`:
  ```json
  "build:capacitor": "CAPACITOR_BUILD=1 tsc -b && vite build",
  "cap:sync": "npm run build:capacitor && npx cap sync android"
  ```

  Update `CLAUDE.md` commands section:
  ```bash
  npm run cap:sync   # Build for Capacitor and sync to Android project
  ```

  - **Manual tests:** Run `npm run build:capacitor` — verify the `dist/` output contains no service worker registration. Run `npm run build` (no env var) — verify service worker is still registered in the normal web build.

- [ ] **A.4 — Gitignore Android build artifacts**

  The `web-client/android/` folder generated by Capacitor should be committed (it contains native config files and Gradle wrappers that are edited over time). Build artifacts inside it should not.

  Add to `web-client/.gitignore` (or root `.gitignore`):
  ```
  web-client/android/.gradle/
  web-client/android/app/build/
  web-client/android/build/
  web-client/android/local.properties
  ```

  Commit `web-client/android/` (minus the above). This makes the Android project reproducible without `npx cap add android` on every clone.

- [ ] **A.5 — Safe area insets for status bar and gesture nav bar**

  Android devices overlay the WebView with a status bar (top) and a gesture nav bar (bottom). Without handling, content slides under them.

  In `web-client/index.html`, update the viewport meta tag:
  ```html
  <meta name="viewport" content="width=device-width, initial-scale=1.0, viewport-fit=cover" />
  ```
  `viewport-fit=cover` is required for `env(safe-area-inset-*)` to work.

  In `web-client/src/index.css`, add global safe area padding to the app root:
  ```css
  /* Push content below the native status bar on Capacitor */
  #root {
    padding-top: env(safe-area-inset-top);
  }
  ```

  In `web-client/src/components/calorie-log/FloatingActionButton.tsx`, offset the FAB above the gesture nav bar. The current `bottom-6` class needs to account for the inset:
  ```tsx
  style={{ bottom: 'calc(1.5rem + env(safe-area-inset-bottom))' }}
  ```
  Replace the Tailwind `bottom-6` with this inline style (or a CSS variable approach).

  - **Manual tests:** Install on Pixel. Verify: (1) sticky header is not hidden behind the status bar, (2) FAB is not hidden behind the Android gesture nav bar, (3) the bottom of the page content is not clipped.

### Phase B: Platform Abstraction

- [ ] **B.1 — Create `src/platform/index.ts`**

  Single file that all components import for platform detection. No component imports `@capacitor/core` directly — this is the only callsite.

  Create `web-client/src/platform/index.ts`:
  ```ts
  import { Capacitor } from '@capacitor/core'

  /* Platform flags — checked once at startup, stable for the app lifetime. */
  export const platform = {
    // True when running inside a Capacitor native wrapper (Android/iOS).
    // False in the browser (web or PWA).
    isNative: Capacitor.isNativePlatform(),

    // Convenience aliases for future native feature gating.
    // Example: if (platform.isAndroid) { await Haptics.impact() }
    isAndroid: Capacitor.getPlatform() === 'android',
  }
  ```

  No tests needed — this is a passthrough to Capacitor's own detection.

- [ ] **B.2 — Hide InlineAddRow on mobile viewports and native**

  `InlineAddRow` is a desktop-only interaction pattern (keyboard-driven, column-aligned inputs). On mobile, the FAB → `AddItemSheet` is the correct flow.

  Two layers of hiding:
  1. **CSS breakpoint** (`hidden sm:table-row`) on InlineAddRow's collapsed `<tr>` — hides it below the `sm` breakpoint in any browser (web or native). This is what Playwright's mobile viewport tests will verify.
  2. **Platform gate** in `MealSection` (`ItemTable.tsx`) — don't render `<InlineAddRow>` at all when `platform.isNative`, regardless of viewport width.

  In `web-client/src/components/calorie-log/InlineAddRow.tsx`, add `hidden sm:table-row` to the collapsed trigger `<tr>`:
  ```tsx
  <tr className="border-t border-gray-50 hidden sm:table-row">
  ```

  In `web-client/src/components/calorie-log/ItemTable.tsx` in `MealSection`, gate the render:
  ```tsx
  import { platform } from '../../platform'

  // InlineAddRow is a desktop-only interaction — on native, the FAB → sheet is the mobile path.
  {!platform.isNative && (
    <InlineAddRow ... />
  )}
  ```

  - **Vitest component test:** Add a test to `web-client/src/components/calorie-log/__tests__/` that renders `ItemTable` with `platform.isNative = true` (mock the module) and asserts `InlineAddRow` is not in the DOM.
  - **Manual tests:** On Pixel — verify no inline add rows appear. Verify FAB opens AddItemSheet and an item can be added successfully.

### Phase C: Playwright E2E Enhancement

- [ ] **C.1 — Add data-testid attributes to key interactive elements**

  Current E2E tests use CSS class selectors like `button.fixed.bottom-6.right-6` which are fragile — a Tailwind refactor breaks the test for reasons unrelated to behaviour. Replace with stable `data-testid` attributes.

  Add `data-testid="fab-add"` to the FAB button in `web-client/src/components/calorie-log/FloatingActionButton.tsx`.

  Add `data-testid="add-item-sheet"` to the sheet root in `web-client/src/components/calorie-log/AddItemSheet.tsx`.

  Update `e2e/tests/calorie-log.spec.ts` to use `page.getByTestId('fab-add')` instead of the CSS class selector. All existing tests should still pass after the swap.

  - **E2E tests:** Run the full existing E2E suite (`npx playwright test`) and confirm no regressions.

- [ ] **C.2 — Add Mobile Chrome project to Playwright config**

  In `e2e/playwright.config.ts`, add a second project:
  ```ts
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
    { name: 'Mobile Chrome', use: { ...devices['Pixel 7'] } },
  ],
  ```

  The `Pixel 7` device preset uses a ~412px wide viewport, which falls below the `sm` (640px) breakpoint — so responsive hiding will be exercised automatically on all existing tests.

- [ ] **C.3 — Add mobile-specific calorie log smoke tests**

  Create `e2e/tests/calorie-log-mobile.spec.ts` with a `test.use({ ...devices['Pixel 7'] })` block. These tests verify mobile-specific behaviour that the desktop suite doesn't cover:

  - `InlineAddRow "+ Add" trigger is not visible` — assert the collapsed `+ Add` button is hidden on narrow viewport (verifies B.2 CSS gate).
  - `FAB is visible and opens AddItemSheet` — tap FAB, assert sheet appears.
  - `Add item via FAB → sheet → item appears in list` — full happy path on mobile viewport.

  Reuse the same `beforeEach` login pattern from `calorie-log.spec.ts`.

  - **E2E tests:** Run `npx playwright test --project="Mobile Chrome"` to confirm these pass in isolation.
